## 摘 要

手语是听障人士沟通的重要方式，然而日常生活中大多数人并不理解手语，导致听障群体在交流中面临巨大障碍。本项目拟设计并实现一套基于树莓派的实时手语识别系统：以树莓派4B作为嵌入式识别终端，通过CSI摄像头实时采集手语动作视频，利用MediaPipe Holistic模型提取人体姿态及手部关键点坐标，采用LSTM时序神经网络对连续手语动作进行分类识别，并通过OpenCV在本地显示器上实时展示识别结果。

本系统拟采用树莓派4B作为嵌入式主控平台，搭载CSI摄像头模块进行视频采集；使用MediaPipe Holistic框架提取人体33个姿态关键点和双手各21个手部关键点（共计258维特征向量），以30帧滑动窗口构建时序输入序列；在识别算法层面，拟采用三层LSTM网络（64→128→64隐藏单元）结合两层全连接层（64→32→Softmax）作为时序分类模型，使用TensorFlow Keras框架在PC端完成训练后导出H5模型部署至树莓派端侧推理；树莓派端使用OpenCV构建图形化界面，在窗口中同时展示摄像头实时画面（叠加MediaPipe骨架连线）、各手势类别概率柱状图、当前识别词汇及完整手语句子，实现听障人士与普通人群之间的无障碍沟通。

**关键词：** 树莓派；手语识别；MediaPipe Holistic；LSTM；TensorFlow Keras；OpenCV

## 第1章 引言

### 1 研究背景

根据世界卫生组织统计，全球约有4.3亿人患有不同程度的听力障碍，中国约有2780万听障人士，位居各类残疾之首。手语作为听障群体的第一语言，是他们表达思想、获取信息、参与社会活动的重要工具。然而，普通人群中掌握手语的比例极低（不足1%），这导致听障人士在就医、购物、办事等日常生活场景中面临严重的沟通障碍，形成了巨大的"信息鸿沟"。

近年来，随着深度学习技术的飞速发展，计算机视觉在人体姿态估计、动作识别等领域取得了突破性进展。Google开源的MediaPipe Holistic框架提供了跨平台、高效的全身姿态及手部关键点联合检测方案，可在一帧图像中同时提取人体33个姿态关键点和双手各21个手部关键点的三维坐标[1]。循环神经网络（RNN）及其变体LSTM在时序数据建模方面表现优异，已成功应用于手语识别、行为识别等序列分类任务[5]。在嵌入式平台方面，树莓派作为一款低成本、高性能的单板计算机，搭载ARM Cortex-A系列处理器，具备运行深度学习推理框架的能力，配合专用摄像头模块，为嵌入式端侧手语识别提供了理想的硬件平台。

YouTube创作者Nicholas Renotte已开源了一套基于MediaPipe Holistic + LSTM的手势识别完整方案[2]，使用TensorFlow Keras搭建三层LSTM网络，通过OpenCV窗口展示实时识别画面与概率分布，在GitHub上已获数千星标。国内开发者kalakalabala在CSDN上发表了该方案的实战教程[3]，详细介绍了使用MediaPipe导出身体节点坐标、搭建LSTM网络训练手势检测模型并成功部署到树莓派4B的完整流程——训练了"666"、"thumbs_up"、"finger_heart"、"scissor_hand"四种手势的识别模型，在树莓派4B上运行成功，验证了该技术路线在嵌入式平台上的可行性。本项目将在此方案基础上进行复现与扩展：一是参考kalakalabala的实战流程，完整复现MediaPipe Holistic关键点提取→数据集采集→LSTM模型训练→树莓派部署的完整管线；二是在原方案4种手势的基础上扩展至更丰富的手语词汇集，提升系统的实际应用价值。

### 2 研究分析

#### 1 可行性分析

**（1）技术可行性**

本系统核心技术栈包括姿态与手部关键点检测、时序动作分类以及OpenCV界面开发三个方面，各方向均有成熟的开源框架和社区支持：

- **姿态与手部关键点检测**：MediaPipe Holistic模型可在单帧图像中同时检测人体33个姿态关键点（每点含x, y, z, visibility 4个值）和双手各21个手部关键点（每点含x, y, z 3个值），共计258维特征向量。参考kalakalabala的方案[3]，不使用面部网格（468个节点），仅保留姿态和手部坐标，有效降低特征维度。该模型已在超过30万张标注图像上训练，在树莓派4B上推理速度可达15-20 FPS，满足实时处理需求[1]。
- **时序动作分类**：参考Nicholas Renotte的开源方案[2]及kalakalabala的实战验证[3]，采用三层LSTM网络进行手语动作的时序分类，该方案已在树莓派4B上验证可行。模型参数规模适中（约50万可训练参数），在自采集的小规模手语数据集上训练2000轮即可收敛，单次推理可在树莓派CPU上完成（约50-100ms），满足实时性要求。
- **OpenCV界面**：树莓派端使用OpenCV（cv2）构建图形化界面——通过`cv2.imshow`创建显示窗口，利用`cv2.rectangle`、`cv2.putText`等函数在帧上绘制骨架叠加、概率条和识别文本，与参考方案[2][3]的`prob_viz`函数实现方式完全一致，技术成熟可靠。树莓派可通过HDMI接口连接显示器进行本地显示。

**（2）经济可行性**

整套硬件成本估算如下：树莓派4B（4GB版）约350元 + CSI摄像头模块约60元 + MicroSD卡约30元 + 电源及配件约80元 = **总计约520元**。若需本地显示，可连接现有显示器（利用现有设备），无需额外购置显示屏。所有软件均为开源方案（MediaPipe、TensorFlow、OpenCV），无需额外授权费用，整体成本可控。

**（3）操作可行性**

系统设计目标为树莓派上电自动运行识别程序（配置systemd自启动），用户仅需面对树莓派摄像头做手语动作，即可在显示器上实时看到识别结果。操作方式为按`Q`键退出程序，简单直观。

#### 2 国内外应用现状

近年来，手语识别技术受到学术界和产业界的广泛关注。在国际上，Google的MediaPipe团队已开源了手部关键点检测与全身姿态估计的完整方案[1]；Nicholas Renotte在YouTube上发布了基于MediaPipe Holistic和LSTM的手势识别完整教程，并在GitHub开源了全部代码，已获数千星标[2]。国内方面，kalakalabala在CSDN上发表了该方案的详细实战教程，介绍了从环境搭建、数据集采集、模型训练到树莓派4B部署的完整流程[3]；中科院计算所、清华大学等机构在手语数据集构建和连续手语识别方面有较多研究成果[4]，科大讯飞、腾讯等企业也在无障碍AI方向有所布局。但目前面向嵌入式终端的实时手语识别产品仍较为稀缺——现有方案多依赖高性能GPU服务器或昂贵的深度摄像头（如Kinect），而基于树莓派等低成本嵌入式平台实现端侧识别并本地实时显示完整手语句子的方案尚未见成熟应用。本项目的定位正是填补这一空白。

## 第2章 项目设计方案

### 2.1 软件设计方案

本系统软件部分采用**树莓派单机部署**架构，所有模块（视频采集、关键点提取、LSTM推理、界面显示）均运行于树莓派4B之上。模型训练阶段在PC端完成（利用GPU加速），训练好的H5模型文件拷贝至树莓派进行端侧推理。系统分为四个核心模块：**视频采集模块**、**关键点提取模块**、**LSTM识别推理模块**和**OpenCV界面显示模块**。

#### 2.1.1 视频采集模块

使用OpenCV（cv2）调用树莓派CSI摄像头，以640×480分辨率、30FPS帧率采集视频流。该模块负责帧数据的获取、格式转换（BGR→RGB，适配MediaPipe输入要求）及帧率控制。

核心处理流程（参考kalakalabala方案[3]中的`mediapipe_detection`函数）：
- 调用`cv2.VideoCapture(0)`打开CSI摄像头；
- 每帧通过`cv2.cvtColor(image, cv2.COLOR_BGR2RGB)`进行颜色空间转换；
- 设置`image.flags.writeable = False`（优化内存，提升MediaPipe推理效率）；
- 传入MediaPipe Holistic模型进行推理；
- 推理完成后恢复`image.flags.writeable = True`，并通过`cv2.cvtColor(image, cv2.COLOR_RGB2BGR)`转回BGR格式，用于后续骨架绘制和显示。

#### 2.1.2 关键点提取模块

采用**MediaPipe Holistic**模型，对每帧图像同时进行人体姿态、左手和右手的联合关键点检测。参考kalakalabala方案[3]，不使用面部网格（468个节点），仅保留姿态和手部坐标，每帧提取的特征向量维度如下：

| 检测目标 | 关键点数量 | 每点数据 | 向量长度 |
|---------|-----------|---------|---------|
| 人体姿态（Pose） | 33个 | x, y, z, visibility | 33 × 4 = 132 |
| 左手（Left Hand） | 21个 | x, y, z | 21 × 3 = 63 |
| 右手（Right Hand） | 21个 | x, y, z | 21 × 3 = 63 |
| **合计** | — | — | **258** |

参考Nicholas Renotte方案[2]及kalakalabala实现[3]中的`extract_keypoints`函数，对每帧提取258维特征向量，代码逻辑如下：

```python
def extract_keypoints(results):
    pose = np.array([[res.x, res.y, res.z, res.visibility] for res in
              results.pose_landmarks.landmark]).flatten() \
              if results.pose_landmarks else np.zeros(33*4)
    # 不使用面部网格（face），注释掉
    # face = np.array([[res.x, res.y, res.z] for res in
    #           results.face_landmarks.landmark]).flatten() \
    #           if results.face_landmarks else np.zeros(468*3)
    lh = np.array([[res.x, res.y, res.z] for res in
              results.left_hand_landmarks.landmark]).flatten() \
              if results.left_hand_landmarks else np.zeros(21*3)
    rh = np.array([[res.x, res.y, res.z] for res in
              results.right_hand_landmarks.landmark]).flatten() \
              if results.right_hand_landmarks else np.zeros(21*3)
    return np.concatenate([pose, lh, rh])
```

若某模态（如手部）在当前帧未被检测到，则对应位置填充零向量作为占位。同时，维护一个长度为30帧的滑动窗口队列（`sequence`列表），始终保持最近30帧的关键点数据，用于后续LSTM模型的输入。

在每帧完成关键点提取后，调用`draw_styled_landmarks`函数[2][3]在BGR帧上绘制骨架叠加：姿态关键点连线使用深棕色（RGB(80,22,10)）和紫色（RGB(80,44,121)），左手连线使用紫色（RGB(121,22,76)）和浅紫色（RGB(121,44,250)），右手连线使用橙色（RGB(245,117,66)）和粉色（RGB(245,66,230)）。

#### 2.1.3 LSTM识别推理模块

这是本系统的核心算法模块，参考Nicholas Renotte的开源方案[2]及kalakalabala的实战实现[3]，拟采用**三层LSTM + 两层全连接**的网络架构：

**表2-1 LSTM网络结构**

| 层序号 | 层类型 | 参数配置 | 说明 |
|-------|--------|---------|------|
| 输入层 | Input | (30, 258) | 30帧 × 258维/帧 |
| 第1层 | LSTM | 64单元, return_sequences=True, activation='relu' | 第一层时序特征提取，保留完整序列 |
| 第2层 | LSTM | 128单元, return_sequences=True, activation='relu' | 第二层时序特征提取 |
| 第3层 | LSTM | 64单元, return_sequences=False, activation='relu' | 第三层时序特征提取，折叠为单向量 |
| 第4层 | Dense | 64单元, activation='relu' | 全连接特征映射 |
| 第5层 | Dense | 32单元, activation='relu' | 全连接特征映射 |
| 输出层 | Dense | N类（手语词汇数量）, activation='softmax' | 多分类概率输出 |

模型训练阶段在PC端使用TensorFlow Keras框架完成（支持GPU加速训练），具体训练流程参照kalakalabala的实践[3]：

1. **数据集采集**：使用数据采集脚本，对每种手语动作，采集30个序列（`no_sequences=30`），每个序列30帧（`sequence_length=30`），每帧保存为`.npy`格式的关键点坐标文件。数据目录结构为`MP_Data/{动作名称}/{序列编号}/{帧编号}.npy`。**需要特别注意**：MediaPipe提取的关键点坐标会随使用者与摄像头的距离和角度变化而变化，因此在数据集采集阶段与实际推理阶段，使用者与摄像头的相对位置、手势动作幅度必须保持基本一致，否则识别准确率将大幅下降[3]。数据采集时程序会在每条序列开始前显示"STARTING COLLECTION"提示，并等待2秒让使用者做好准备。

2. **数据加载与预处理**：通过遍历目录结构加载所有`.npy`文件，构建形状为`(样本数, 30, 258)`的输入数组`X`和对应的标签数组`y`；使用`to_categorical`对标签进行one-hot编码；按95%:5%的比例使用`train_test_split`划分训练集与测试集。

3. **模型编译与训练**：使用Adam优化器，分类交叉熵损失函数（`categorical_crossentropy`），以分类准确率（`categorical_accuracy`）为评估指标；训练2000轮（`epochs=2000`），同时配置TensorBoard回调记录训练日志。参考kalakalabala的实践[3]，即使在CPU上训练2000轮，速度也很快。

4. **模型导出与部署**：训练完成后将模型保存为`action.h5`文件，拷贝至树莓派SD卡中供端侧推理使用。在树莓派端加载模型时，需先使用Keras重建与训练时完全相同的网络结构，然后调用`model.load_weights('action.h5')`加载权重[3]。

在树莓派端的实时推理流程：
- 维护30帧滑动窗口队列（`sequence`列表），每新增一帧关键点数据即移除最旧的一帧（`sequence = sequence[-30:]`）；
- 当队列累积满30帧后，调用`model.predict(np.expand_dims(sequence, axis=0))`进行一次推理；
- 输出各类别的概率分布（shape为`(1, N)`的数组），取最大概率类别作为当前识别结果；
- 设置置信度阈值（`threshold=0.8`），仅当最大概率超过阈值时才采纳本次识别结果。

**手语句子拼接逻辑：** 系统维护一个句子列表`sentence=[]`，用于存储已确认的手语词汇序列。参考Nicholas Renotte方案[2]及kalakalabala实现[3]中的去重逻辑：仅当新识别词与`sentence`中最后一个词不同时才追加（`if actions[np.argmax(res)] != sentence[-1]: sentence.append(...)`），以避免同一手势在连续多帧中被重复计入。设置句子最大长度为5个词汇（`sentence = sentence[-5:]`），超过时自动截断保留最近5个词。

#### 2.1.4 OpenCV界面显示模块

树莓派端使用OpenCV构建图形化界面，通过`cv2.imshow`创建单一显示窗口，利用`cv2.rectangle`、`cv2.putText`等函数在摄像头帧上直接叠加绘制骨架连线、概率条和识别文本，与参考方案[2][3]中的`prob_viz`函数实现方式完全一致。所有信息（视频画面、骨架、概率条、句子、识别结果）均通过图层叠加方式合成在同一帧上，通过HDMI接口输出至本地显示器。详细界面设计见2.3节。

*图2-3 OpenCV图形化界面布局设计（单窗口叠加显示，与参考方案[2][3]一致）*

界面各区域技术实现说明：

**（1）视频画面区（主画面）**

- 摄像头采集的原始帧叠加MediaPipe骨架连线后作为背景显示；
- 骨架采用参考方案的配色：姿态关键点连线为深棕色（RGB(80,22,10)）和紫色（RGB(80,44,121)），左手连线为紫色（RGB(121,22,76)）和浅紫色（RGB(121,44,250)），右手连线为橙色（RGB(245,117,66)）和粉色（RGB(245,66,230)）[2][3]。

**（2）概率柱状图区（画面左侧叠加）**

完全参考Nicholas Renotte方案[2]及kalakalabala实现[3]的`prob_viz`函数，在视频帧上直接叠加绘制，无需额外窗口或画布：

```python
def prob_viz(res, actions, input_frame, colors):
    output_frame = input_frame.copy()
    for num, prob in enumerate(res):
        cv2.rectangle(output_frame,
                      (0, 60 + num*40),
                      (int(prob*100), 90 + num*40),
                      colors[num], -1)   # 概率条（宽度 = 概率×100像素）
        cv2.putText(output_frame,
                    actions[num],
                    (0, 85 + num*40),
                    cv2.FONT_HERSHEY_SIMPLEX, 1, (255,255,255), 2, cv2.LINE_AA)
    return output_frame
```

每种手势分配不同颜色（橙色(245,117,16)、绿色(117,245,16)、蓝色(16,117,245)等），概率条越长代表模型对该类别的置信度越高，直观展示分类判断倾向。

**（3）完整句子显示区（画面顶部）**

- 在画面顶部使用`cv2.rectangle()`绘制橙色背景矩形条（`(245, 117, 16)`填充，高度40像素，宽度640像素）；
- 内部使用`cv2.putText()`以`FONT_HERSHEY_SIMPLEX`字体（字号1.0，白色文字）逐词显示当前句子列表中的词汇，以空格分隔形成完整句子；
- 句子过长时自动截断显示最后5个词（与`sentence[-5:]`逻辑一致[2][3]）。

**（4）当前识别显示区**

- 使用`cv2.putText()`在画面中绘制当前识别到的词汇及置信度百分比；
- 仅当预测概率超过阈值（`threshold=0.8`）时采纳识别结果并更新显示。

**（5）键盘交互控制**

所有控制操作通过键盘按键实现（与参考方案[2][3]一致）：

| 按键 | 功能 | 实现方式 |
|------|------|---------|
| `Q` 键 | 退出程序 | 关闭OpenCV窗口，释放摄像头和模型资源 |

键盘事件通过`cv2.waitKey(10) & 0xFF`捕获，判断按键值是否为`ord('q')`，与参考方案[2][3]的实现方式完全一致。

**最终合成帧尺寸为640×480像素**，所有信息（骨架、概率条、句子、识别文本）均通过OpenCV绘图函数直接叠加在原始摄像头帧上，最终通过`cv2.imshow('OpenCV Feed', image)`在单一窗口中显示。这一设计与参考方案[2][3]完全一致，技术简单可靠，无需额外的GUI框架或多窗口管理。

## 第3章 方案论证及设备选型

### 3.1 方案论证

#### 3.1.1 为何选择树莓派而非Arduino/STM32/ESP32

手语识别系统的核心计算任务包括两个阶段：MediaPipe Holistic关键点检测和LSTM时序推理，两者均涉及大量浮点矩阵运算。Arduino Uno的ATmega328P为8位MCU，主频16MHz，无浮点运算单元（FPU），完全无法承载此类计算；STM32虽具备ARM Cortex-M系列处理器（如STM32F4主频168MHz），但算力（约200 DMIPS）仍远不足以运行MediaPipe模型；ESP32虽有WiFi能力但算力更低（约600 DMIPS）。树莓派4B搭载ARM Cortex-A72四核处理器，算力超过5000 DMIPS，支持完整的Linux操作系统（Raspberry Pi OS），可直接安装TensorFlow 2.x和MediaPipe Python库，且kalakalabala已在树莓派4B上成功运行了完整的MediaPipe Holistic + LSTM实时推理管线[3]，验证了该平台的技术可行性。

#### 3.1.2 为何选择MediaPipe Holistic而非仅使用MediaPipe Hands

参考Nicholas Renotte方案[2]及kalakalabala实践[3]，本项目选择MediaPipe Holistic同时提取人体姿态和双手关键点，而非仅使用MediaPipe Hands单独检测手部。理由如下：

| 对比维度 | MediaPipe Hands（仅手部） | MediaPipe Holistic（全身+双手） |
|---------|-------------------------|-------------------------------|
| 特征维度 | 21点×3=63维/手 | 33×4 + 21×3×2 = 258维 |
| 上肢关联 | 无 | 姿态关键点包含肩、肘、腕，可与手部形成空间关联 |
| 鲁棒性 | 手部快速移动时易丢失 | 姿态检测辅助定位手部大致区域，减少跟丢 |
| 手语表达 | 仅捕捉手指动作 | 同时捕捉手臂运动轨迹、身体姿态——手语中许多词汇依赖手臂位置和运动方向（如"谢谢"需手臂画弧、"家"需双手搭屋顶形） |

此外，kalakalabala的实践表明[3]，不使用面部网格（468个节点）仅保留姿态和手部坐标（258维），既能有效降低特征维度、减小模型规模，又不影响手势识别的准确率。

#### 3.1.3 为何选择LSTM而非3D-CNN或Transformer

手语识别本质上是时序动作分类问题。常见方案对比：

| 方案 | 优点 | 缺点 | 适用性 |
|------|-----|------|-------|
| 3D-CNN（如I3D） | 端到端学习原始视频帧 | 计算量大（需处理RGB像素级数据）、内存消耗高 | 需GPU，不适合树莓派 |
| **LSTM（本方案）** | 针对时序建模设计、输入为轻量关键点坐标（258维/帧而非整帧图像）、参数量可控 | 需关键点预提取步骤 | **已在树莓派4B上验证可行[2][3]** |
| Transformer | 长程依赖建模能力强、可并行训练 | 推理计算复杂度O(n²)、参数量大 | 暂不适合嵌入式端侧 |

LSTM网络以258维关键点坐标序列作为输入，数据量极低（30帧×258维≈8KB），远小于处理30帧原始图像（30×640×480×3≈27MB）。三层LSTM模型参数量约50万，H5模型文件大小约2MB，在树莓派4B的CPU上单次推理耗时约50-100ms，满足实时性要求。kalakalabala的实践[3]表明即使在CPU上训练2000轮也很快。

#### 3.1.4 为何采用树莓派单机部署架构

本系统将视频采集、关键点提取、LSTM推理、OpenCV显示全部部署于树莓派4B单机运行，而非采用"树莓派+PC"的分布式方案，理由如下：

- **部署简单**：所有模块运行于同一设备，无需处理网络通信、数据同步等分布式系统问题，代码结构清晰、维护方便。参考kalakalabala的描述——"代码和权重文件拖过去运行就好了"[3]；
- **独立运行**：树莓派连接显示器和摄像头即可独立工作，不依赖PC或其他外部设备，适合在各种场合快速部署使用；
- **成本更低**：仅需树莓派+摄像头+电源，连接现有显示器即可，无需额外购置专用显示屏或依赖PC；
- **实时性保证**：所有处理在本地完成，无网络传输延迟，识别结果即时显示；
- **开发门槛低**：参考方案[2][3]本身就是单机架构，本项目在其基础上直接扩展，降低了开发难度和技术风险。
